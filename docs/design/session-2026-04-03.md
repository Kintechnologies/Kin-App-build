## 🔴 Architectural Flag: Navigation Structure Mismatch

**This is the most important finding of this session. It requires a product decision before the next engineering sprint begins.**

### What the v0 spec requires
Three tabs. No more.
```
[ Today ]   [ Conversations ]   [ Settings ]
```
Every domain (meals, budget, fitness, home management, pets) is delivered through intelligence — via the Today screen briefing, alerts, check-in cards, and Conversations — not through dedicated tabs.

**Explicitly out of scope per v0 spec:** Meal planning tab, budget tab, fitness tab, pet profiles tab, family tab as a standalone screen.

### What the current mobile app has
Seven tabs in `_layout.tsx`:
```
index (Today/Dashboard) | chat | meals | budget | family | fitness | settings
```

The current home screen (`index.tsx`) also surfaces: meals cards, grocery reminders, budget spent/total, last chat topic, and quick-action buttons to meals/budget/chat tabs. This is a dashboard pattern, not the intelligence-forward briefing pattern.

### The gap
The existing mobile app was built for the previous product vision (multi-tab Family OS). The new v0 spec is a deliberate, scoped reduction: fewer screens, deeper intelligence, one winning moment first.

This is not a polish gap. It's a structural rebuild. The three-tab v0 app and the seven-tab current app are different products.

### What this means for engineering
**Do not proceed with device testing (task #11) as a validation of the v0 product** — the current app is not the v0 product. Device testing can proceed as a smoke test of the existing build, but the findings will not map to the v0 spec.

The Lead Engineer will need to:
1. Restructure `_layout.tsx` to 3 tabs (Today, Conversations, Settings)
2. Rebuild `index.tsx` as the Today screen (briefing hero, alerts, schedule, check-in cards)
3. Restructure `chat.tsx` as the Conversations screen (thread list + conversation view)
4. Rebuild `settings.tsx` with the correct section order per spec
5. Remove or archive: `meals.tsx`, `budget.tsx`, `family.tsx`, `fitness.tsx` (these are v1+)

This is significant work. Austin should make the call on timing — does this happen this sprint, or does the current build go to TestFlight as a precursor before restructuring?

**Design recommendation:** Rebuild to v0 spec before TestFlight if possible. The risk of shipping the wrong architecture and having users form expectations around it is higher than the risk of a delayed TestFlight.

## Outputs This Session

### 1. Today Screen Component Spec
**File:** `docs/design/today-screen-spec.md`
**Status:** Ready for engineering

Covers:
- Header (greeting, time-aware, date)
- MorningBriefingCard (hero, all states: loading / empty / failed / content)
- ActiveAlertCard (standard urgency vs. high urgency, dismiss behavior, queue logic)
- TodayScheduleSection (calendar connected / not connected / loading / empty / events)
- KinCheckInCards (0–2 cards, dismiss, domain context)
- Screen-level state model
- Navigation map
- All static copy
- Full spacing reference

Key design decisions made:
- Briefing card has no icon, no "AI" branding — the text speaks for itself. A quiet `BRIEFING` label is the only label.
- Alert card uses a left color border (amber/rose), not a full-color background — keeps the screen calm
- Schedule uses color bars by owner (green = user, blue = partner, amber = kid) — no avatars, no initials
- Check-in cards render nothing when empty — no "all clear" filler

### 2. Conversations Screen Component Spec
**File:** `docs/design/conversations-screen-spec.md`
**Status:** Ready for engineering

Covers:
- List view: header, ConversationRow, PinnedSection (household thread), empty state
- Conversation view: header, MessageBubble (user + Kin), MessageInput, KinTypingIndicator
- Quick reply chips (new conversations only, time/context-aware)
- Navigation model
- All static copy
- Full spacing reference

Key design decisions made:
- Household thread is visually distinguished by amber left bar only — no label, no icon in the row itself (a small `"HOUSEHOLD"` section label sits above it)
- No Kin avatar or "Kin" name prefix on messages — position (left-aligned) is the signal
- Quick reply chips populate input but don't auto-send — user confirms before sending
- Chips are context-aware by time of day + coordination state, not generic feature shortcuts

## Tone Review — Current App vs. Intelligence Engine

One observation from reviewing the current `index.tsx` and `chat.tsx` in the context of the voice spec:

The current app has conversation ideas chips labeled: `"Plan this week's meals"`, `"Budget check-in"`, `"High-protein snack ideas"`, `"Kids and activities"`. These are feature-framing prompts, not coordination-intelligence entry points.

Per the intelligence engine §8 (tone rules) and §19 (structural consistency): Kin does not offer a menu of domains. It surfaces what's relevant to the household, right now.

When the Conversations screen is rebuilt per the spec, quick reply chips should be time/context-aware coordination prompts — not a feature index.
