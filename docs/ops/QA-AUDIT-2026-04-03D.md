## Issues Found

### P1 — Must Fix Before TestFlight

---

**B18 — P1 | `apps/web/src/lib/pickup-risk.ts` lines 226–228, 231–233**
**Alert content templates are multi-sentence — violates §8**

Both generated alert strings are two sentences, not one:

- **RED** (line 226–228):
  `"${childName} needs pickup from ${activityName} at ${pickupTimeStr} — both of you have conflicts. Who's covering this?"`
  Two sentences: a statement + a question.

- **YELLOW** (line 231–233):
  `"${childName} needs pickup from ${activityName} at ${pickupTimeStr} — you're in a conflict. Partner is clear if they can cover."`
  Two sentences separated by a period.

Spec §8: "Alerts are exactly 1 sentence: [What changed] — [Implication]"

The question "Who's covering this?" and the follow-on "Partner is clear if they can cover." are separate sentences. The spec format is a single clause with an em-dash separating the event from its implication — everything must land in that one sentence.

**Fix direction:** Collapse each into one sentence:
- RED: `"[Child] needs pickup from [activity] at [time] — both parents have conflicts and no coverage is confirmed."`
- YELLOW: `"[Child] needs pickup from [activity] at [time] — you're in a conflict, partner is free."`

---

**B19 — P1 | `apps/web/src/lib/pickup-risk.ts` line 233**
**YELLOW alert uses conditional qualifier on confirmed data — violates §23**

Line 233: `"Partner is clear if they can cover."`

The `if they can cover` conditional introduces uncertainty. But the detection code has already confirmed at this point that `!partnerBusy` is true — the partner has no calendar conflict in the pickup window. The qualifier is not warranted by the data.

Spec §23: "High confidence outputs: no qualifiers, direct statement."

When the partner is confirmed free, the output should state that directly. The conditional also reads as hedged advice rather than a coordination signal.

**Fix direction:** Remove the conditional. "Partner is free." (or fold into the single-sentence B18 fix above, e.g., `"…you're in a conflict, partner is free."`).

Note: B18 and B19 can be fixed together in a single edit to `pickup-risk.ts`.

---

**B20 — P1 | `apps/web/src/app/api/morning-briefing/route.ts` line 119**
**Coordination issues query uses `profileId` directly — partner users receive no coordination context**

Line 119:
```ts
supabase
  .from("coordination_issues")
  .select("trigger_type, content")
  .or(`household_id.eq.${profileId}`)
  .eq("state", "OPEN")
```

`profileId` here is `user.id` — the ID of whoever is calling the briefing endpoint. Coordination issues are inserted in `pickup-risk.ts` with `household_id: primaryId`, where `primaryId` is always the primary parent's profile ID (resolved by the `household_id` normalization logic in `pickup-risk.ts` lines 103–113).

If the calling user is a **partner** (secondary parent), their `user.id` differs from the `primaryId` stored on the `coordination_issues` row. The query returns zero rows. That partner's morning briefing will never include pickup risk or any coordination context — even when those issues exist for their household.

This is a new issue introduced in Run G (the `openIssues` query was added this session).

**Fix direction:** Resolve the household's `primaryId` before querying — same pattern as `pickup-risk.ts` lines 103–113. Query `household_id.eq.${primaryId}` after resolving whether the current user is primary or partner.
