**[P1-4] `apps/mobile/app/(tabs)/family.tsx` — No error state UI on load failure**
- **Location:** `loadFamilyMembers()` catch block at line 221
- **Issue:** When `loadFamilyMembers()` fails (e.g., network error, Supabase down), the catch block only calls `console.error` and sets `loading = false`. The user sees a blank screen with two empty sections and no indication something went wrong. No retry affordance.
- **Spec requirement:** "All sections from the v0 product spec are present / No missing states (loading, empty, error)"
- **Fix:** Add an `error` state variable. Set it in the catch block. Render an error card with a "Try again" button (mirrors the pattern in meals.tsx `errorCard` style).

### 🔵 P2 — Nice to Fix

---

**[P2-1] `apps/mobile/app/(tabs)/meals.tsx:513` — Dead code: `getGroceryByStore()`**
- **Location:** Lines 514–525
- **Issue:** Function is explicitly commented "Legacy fallback (unused when synced items are present)" and is never called anywhere in the component. With the Realtime C2 path always active, this function is dead code.
- **Fix:** Remove the function. Reduces cognitive overhead for future readers.

---

**[P2-2] Morning briefing system prompt — potential output-limit overrun**
- **Location:** `apps/web/src/app/api/morning-briefing/route.ts` line 265
- **Issue:** The system prompt instructs Claude to "Cover: schedule/commute, calendar conflicts, kids' pickups/activities, budget status, pet care." That's 5 topic categories. The intelligence engine spec sets the bar at **max 1 primary coordination insight + 1 supporting detail** for the morning briefing.
- **Risk:** On a busy family day with multiple active topics, Claude may produce a wall of text rather than a surgical briefing. The product's own canonical example ("Morning. Leave for the gym by 5:55…") appears to set a higher density precedent than the spec's 1+1 rule suggests. This is a design question more than a bug — but worth a deliberate decision before TestFlight.
- **Recommendation:** Either update the spec to reflect the richer briefing intent, or add a `max_tokens: 150–200` cap (instead of current 500) to enforce brevity structurally.
