**[P0-3] SCOPE: `apps/mobile/app/(tabs)/budget.tsx` — Full budget tracking screen built**
- **Rule violated:** Scope enforcement — "Budget tracking tab" flagged immediately.
- **What's there:** A full-featured budget dashboard with `BudgetView` ("setup" | "dashboard") states, category setup flow using `BUDGET_CATEGORIES` from `@kin/shared`, transaction entry sheet with amount/category/date fields, 3-bucket color coding (`needs`, `wants`, `savings`), and `RawTransactionRow` Supabase joins reading `budget_categories` and transactions tables.
- **Mitigating note:** `budget.tsx` is **not registered** in `apps/mobile/app/(tabs)/_layout.tsx`. The layout defines exactly 3 tabs: `index`, `chat`, `settings` — matching the spec. The file is effectively dead code from a navigation standpoint. However, the file exists in the `(tabs)` directory alongside a `packages/shared/src/budget.ts` with `BUDGET_CATEGORIES` and `BudgetBucket` types — indicating a build-out rather than a stub. This needs Austin's explicit call on disposition: delete the file, or confirm scope supersession per the v2 product vision (same question as P0-1 and P0-2 from Run 1).

### 🟡 P1 — Must Fix Before TestFlight

---

**[P1-5] `apps/mobile/app/(tabs)/index.tsx` — Today's Schedule section missing**
- **Spec requirement:** Product spec §1 "Today Screen" layout explicitly lists **Today's Schedule** as a required section: "Combined view: both parents' events, color-coded by person. Kids' schedule overlaid (school end time, activities). Clean list format — time + event + who. Tappable to see full event detail. Updates in real time as calendars change."
- **What's built:** The Today screen renders three layers: Morning Briefing Card, Alert Cards, and Check-in Cards. There is no schedule list component, no calendar events fetch, and no real-time subscription to calendar data in `index.tsx`.
- **Risk:** TestFlight users will see a Today screen with no schedule view — a core spec requirement. The intelligence engine can surface coordination insights (alerts), but users have no at-a-glance view of their actual day.
- **Fix:** Add a `TodaySchedule` section below Alert Cards: fetch events from the calendar tables for today, render as a time-sorted list with per-person color coding, and add a Supabase Realtime subscription with cleanup.

**[P1-9] `apps/mobile/app/(tabs)/chat.tsx` — Thread list has no loading state**
- **Location:** `loadThreads()` function (lines 112–151), conversation list render (lines 329–426)
- **Issue:** When `loadThreads()` runs on mount, `threads` starts as `[]`. The FlatList renders immediately with an empty `ListFooterComponent` — there is no spinner, skeleton, or loading indicator while the async fetch completes. On a slow connection, the "Recent conversations" label and empty thread area are visible for 1–3 seconds with no indication of load.
- **Spec requirement:** "No missing states (loading, empty, error)" — screen completeness check.
- **Fix:** Add a `threadsLoading` boolean state. Set it `true` before the fetch, `false` in finally. Render a skeleton or `ActivityIndicator` in `ListFooterComponent` while `threadsLoading` is true.

**[P2-5] `apps/mobile/app/(tabs)/index.tsx:139` — Resolved alert closure line deviates from spec relief language**
- **Location:** `AlertCard` component, RESOLVED state render (line 139)
- **Copy:** *"Sorted. I'll let you know if anything changes."*
- **Spec requirement (§8):** The conditional relief phrase is: *"I'll flag it if anything changes."* The phrase "let you know" is softer and vaguer than "flag it" — the spec uses "flag" deliberately to signal a specific Kin behavior (raising an alert), not a passive update.
- **Fix:** Update to: *"Sorted. I'll flag it if anything changes."*

---

**[P2-6] `apps/mobile/app/(tabs)/index.tsx:579` — First-use closing line mixes relief language patterns**
- **Location:** First-use card, `firstUseClosingLine` text (line 579)
- **Copy:** *"I'll keep an eye on it and flag anything that changes."*
- **Spec requirement (§8):** The spec maps relief phrases to specific behaviors — monitoring → *"I'll keep an eye on it."*, conditional → *"I'll flag it if anything changes."* These are separate phrases covering distinct Kin behaviors and should not be combined. The combined version ("keep an eye on it *and* flag anything") blurs the distinction and reads as filler padding rather than a precise behavioral commitment.
- **Fix:** Pick the one that matches what Kin will actually do on first open. Since the first-use context is monitoring (Kin is orienting to the household), *"I'll keep an eye on it."* is cleaner.

---

**[P2-7] `apps/mobile/app/(tabs)/index.tsx:167` — "Heads up" label on alert cards**
- **Location:** `AlertCard` component, OPEN state header (line 167)
- **Copy:** `<Text style={styles.alertOpenLabel}>Heads up</Text>`
- **Spec §8:** "Just a heads up…" is explicitly listed as a prohibited phrase. "Heads up" alone as a UI label is not the same construction, but the spirit of the rule is to avoid hedge language that undermines Kin's direct voice. A coordination alert surfaced because something requires action — labeling it "Heads up" suggests the user may want to take note, rather than that action is required.
- **Recommendation:** Replace with a more direct label that matches the trigger type — e.g., *"Conflict"*, *"Action needed"*, or remove the label and let the alert content lead. This is a design call — not a hard failure, but worth a deliberate decision before TestFlight.
