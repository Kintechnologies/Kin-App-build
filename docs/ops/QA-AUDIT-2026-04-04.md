### P2-1 — `late-schedule-change.ts` line 314: Implication clause is a noun phrase, not present/near-future

**File:** `apps/web/src/lib/late-schedule-change.ts`
**Line:** 314
**Code:**
```javascript
coordinationContent = `Your ${timeStr} just landed — pickup conflict for ${childName}'s ${activityName}.`;
```
**Issue:** The implication clause "pickup conflict for [child's activity]" is a descriptive noun phrase. The spec (§9 / late-schedule-change-spec.md §1) requires the implication clause to use present or near-future verb form: "you've got pickup", "dinner window is gone", "pickup's on your partner." "Pickup conflict for..." describes the problem without telling the user what to do.

Approved patterns for comparison:
- "you've got pickup tonight" ✅
- "you'll need coverage" ✅
- "pickup's on your partner" ✅

**Suggested direction:** `Your ${timeStr} just landed — pickup for ${childName} needs coverage.` (or similar imperative/near-future form)
**Owner:** Lead Eng
**Priority:** P2

### P2-2 — `late-schedule-change.ts`: Briefing-mode path creates no persistent record for morning briefing

**File:** `apps/web/src/lib/late-schedule-change.ts`
**Lines:** 199–202 (early return in briefing mode)
**Issue:** When `deliveryMode() === "briefing"` (before 10am or after 6pm), the function returns 0 without creating any record. The code comment says "the morning briefing route picks up open coordination_issues via its own query" — but no issue is created, so there is nothing to pick up.

The spec (late-schedule-change-spec.md §2) says for the after-6pm mode: "No — **store**, include in next day's briefing context." The implementation does not store anything. After-6pm calendar changes that introduce coordination dependencies are silently dropped; the next morning's briefing will not know about them unless the pickup risk cron separately catches the conflict.

This means the briefing-mode delivery path for late schedule changes is currently a no-op. For pickup conflicts, the morning pickup-risk detection provides a fallback. For evening-window coordination changes (non-pickup), there is no fallback — these are dropped.

**Scope note:** The spec marks after-6pm "queueing" as a delivery mode to implement, not as a Layer 2 deferral. However, given that pickup risk detection provides coverage for the most critical case, and TestFlight scope prioritizes immediate-mode alerts, this may be acceptable as a v0 limitation.

**Owner:** Lead Eng (flag for decision: ship as-is for TestFlight or implement minimal queueing)
**Priority:** P2

### P2-3 — `late-schedule-change.ts` line 335: Evening window alert change clause is vague

**File:** `apps/web/src/lib/late-schedule-change.ts`
**Line:** 335
**Code:**
```javascript
coordinationContent = `Your partner's ${timeStr} just changed — check your evening coverage.`;
```
**Issue:** "just changed" is vague — it does not specify what changed (moved to a later time? cancelled? new event added?). The spec (late-schedule-change-spec.md §1) says the change clause should name the specific change: "just moved to 7:30", "just landed", "just cleared."

For events in the evening coordination window (4pm–8pm) that don't overlap a pickup, the system knows a partner event changed during that window but may not know the nature of the change. If the nature is unknown, this may be an inherent limitation — but "just changed" gives the user less actionable information than "just landed" or "just moved."

**Suggested direction:** Use the event's `updated_at` vs. `created_at` to distinguish "just landed" (new event) from "just moved" (updated event) and produce a more specific change clause.
**Owner:** Lead Eng
**Priority:** P2
