### P2-6 — `generate-alert-content.ts` line 160: No empty-string guard on `parsed.content`

**File:** `apps/web/src/lib/generate-alert-content.ts`
**Line:** 160
**Code:**
```typescript
return { content: parsed.content, severity: parsed.severity };
```
**Issue:** When the AI returns a valid JSON object but with an empty `content` string (e.g., `{content: "", severity: "RED", trigger_type: "PICKUP_RISK"}`), this passes the `if (!parsed)` null-check and returns an empty string as `content`. That empty string is then inserted into `coordination_issues.content` and rendered on the alert card in OPEN state — a blank, visible alert card with no text.

This is an edge case (the model is instructed to return null when uncertain, not empty string), but it is possible under unexpected model behavior and would produce a user-facing regression.

**Required fix:** Add empty-content guard before returning:
```typescript
if (!parsed.content) return { content: fallback, severity: input.severity as "RED" | "YELLOW" };
return { content: parsed.content, severity: parsed.severity };
```
**Owner:** Lead Eng
**Priority:** P2

### P2-7 — `morning-briefing/route.ts`: System prompt omits `INPUT FORMAT` section; diverges from `morning-briefing-prompt.md`

**File:** `apps/web/src/app/api/morning-briefing/route.ts`, line 333 (system prompt)
**Issue:** The `morning-briefing-prompt.md` prompt file specifies an `## INPUT FORMAT` section describing structured JSON fields: `today_events`, `pickup_assignments`, `household_conflicts`, `last_coordination_change`, `current_time`. The system prompt embedded in `route.ts` omits this section entirely. The actual input being passed to the AI is the `briefingContext` text string — a formatted multi-section document, not the JSON structure the prompt file describes.

The system prompt comment says "Source of truth: docs/prompts/morning-briefing-prompt.md" — but the INPUT FORMAT section is not in `route.ts` because the actual input is text. This creates:
1. A false source-of-truth claim (the prompt file describes input the route never sends)
2. Risk that a future IE edit to `morning-briefing-prompt.md` INPUT FORMAT will appear to change the prompt spec without affecting actual AI behavior
3. A maintenance debt: the two versions of the prompt will diverge further over time

**Scope note:** Current behavior is correct — the AI receives the rich text context and produces compliant JSON output. This is not a regression. But it should be resolved before the prompt file becomes IE's canonical reference for TestFlight.

**Recommended resolution:** Either (a) update `morning-briefing-prompt.md` to reflect the actual text input format the route sends (with a note that structured JSON is a future IE direction), or (b) migrate the `briefingContext` to a structured JSON object matching the prompt spec. Option (a) is the v0-correct path.

**Owner:** Intelligence Engineer (prompt spec update) or Lead Eng (route refactor)
**Priority:** P2 — flag for IE when S1.8 drift review is delivered
